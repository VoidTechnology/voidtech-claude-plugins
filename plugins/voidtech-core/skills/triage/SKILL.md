# Triage

把项目 issue 追踪器上的 issue 推过一个由 triage 角色构成的小型状态机。

如果本仓库把外部 pull request 视为一种请求来源（见 issue-tracker 配置），triage 也覆盖它们：**一个 PR 就是一个附带代码的 issue**——相同的角色、相同的状态、相同的状态机，只有少数差异在下文标注为“for a PR”。按追踪器配置把裸 `#42` 解析为一个 issue 或 PR。

triage 期间发布到 issue 追踪器的每一条评论或 issue **都必须**以这段免责声明开头：

```
> *This was generated by AI during triage.*
```

## 角色

两个**category（分类）**角色：

- `bug` — 某处坏了
- `enhancement` — 新功能或改进

五个**state（状态）**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 等待报告者补充更多信息
- `ready-for-agent` — 完整规约，可供 AFK agent 上手
- `ready-for-human` — 需要人工实现
- `wontfix` — 不会处理

对一个 PR，相同的状态是针对其附带的代码来读的：`ready-for-agent` 表示已附上 brief，应由 agent 对该 diff 推进下一步；`ready-for-human` 表示它已准备好由人来合并。

每个被 triage 的 issue 应当恰好携带一个 category 角色和一个 state 角色。如果 state 角色相互冲突，先标出来并询问维护者，然后再做任何其他事。

这些是规范的角色名——issue 追踪器中实际使用的标签字符串可能不同。该映射应当已经提供给你了——如果没有，运行 `/setup-matt-pocock-skills`。

状态流转：一个未打标签的 issue 通常先进入 `needs-triage`；从那里它移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后回到 `needs-triage`。维护者可以随时覆盖——把看起来异常的流转标出来并在继续前询问。

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要什么。解读该请求并采取行动。示例：

- "Show me anything that needs my attention"
- "Let's look at #42"（issue 或 PR）
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 展示需要关注的内容

查询 issue 追踪器并呈现三个桶，最旧的在前：

1. **Unlabeled** — 从未被 triage 过。
2. **`needs-triage`** — 评估进行中。
3. **`needs-info` 且自上次 triage 备注以来有报告者活动** — 需要重新评估。

当 PR 在范围内时，把外部 PR 也纳入这些桶，并给每行打上 `[PR]` 或 `[issue]` 标记。发现（discovery）只浮现*外部* PR（追踪器配置定义谁算外部）——协作者在途的 PR 不是 triage 工作。这个过滤仅用于发现；被显式点名的 PR 总是会被 triage，无论作者是谁。

为每个条目展示计数和一行摘要。让维护者来挑选。

## Triage 某个特定的 issue 或 PR

1. **收集上下文。** 阅读完整的 issue 或 PR（正文、评论、标签、作者、日期；对 PR 还包括 diff）。解析任何既有的 triage 备注，这样你就不会重问已解决的问题。使用项目的领域术语表探查代码库，尊重该区域内的 ADR。对代码库做两项检查：(a) **冗余** —— 按领域概念（而非仅按请求的措辞）搜索是否已有对所请求行为的实现，并报告你查了哪些地方。若找到，它就是一个已实现的 `wontfix`（步骤 5）。(b) **既往拒绝** —— 阅读 `.out-of-scope/*.md` 并浮现任何与此请求相似的记录。

2. **给出建议。** 把你的 category 与 state 建议连同理由告诉维护者，再加上与请求相关的简短代码库摘要——包括它是否已被实现。等待指示。

3. **验证主张。** 在任何盘问之前，先检查主张是否站得住脚。对一个 bug，按报告者的步骤复现它。对一个 PR，确认该 diff 做到了它声称的事——把它 check out，运行相关的测试或命令。报告发生了什么：已确认（附代码路径）、失败，或细节不足（一个强烈的 `needs-info` 信号）。一次已确认的验证能造就强得多的 agent brief。

4. **盘问（如有需要）。** 如果请求需要充实，同时运行 `/grilling` 和 `/domain-modeling` 技能——一次一个问题地把它盘问成型，打磨领域术语，并在决策落地时就地更新 `CONTEXT.md`/ADR。

5. **应用结果：**
   - `ready-for-agent` — 发布一条 agent brief 评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 与 agent brief 结构相同，但注明为什么它不能被委派（判断性决定、外部访问、设计决策、人工测试）。
   - `needs-info` — 发布 triage 备注（模板见下）。
   - `wontfix` — 关闭，评论内容取决于*为什么*：
     - **已实现** — 该变更在代码库中已经存在。指出它所在之处；**不要**写入 `.out-of-scope/`（那个知识库是给*被拒绝的*请求用的，不是给已构建的）。
     - **拒绝（bug）** — 礼貌地解释，然后关闭。
     - **拒绝（enhancement）** — 写入 `.out-of-scope/`，从评论中链接到它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用该角色。如果有部分进展，可选地加评论。

## 快速状态覆盖

如果维护者说 "move #42 to ready-for-agent"，相信他们并直接应用该角色。确认你将要做的事（角色变更、评论、关闭），然后行动。跳过盘问。如果在没有盘问会话的情况下移到 `ready-for-agent`，询问他们是否想写一份 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

把盘问期间解决的一切都记在 "established so far" 之下，这样工作就不会丢失。问题必须具体且可操作，而不是“请提供更多信息”。

## 恢复之前的会话

如果 issue 或 PR 上存在既有的 triage 备注，阅读它们，检查报告者是否回答了任何未决问题，并在继续前呈现一幅更新后的全貌。不要重问已解决的问题。
