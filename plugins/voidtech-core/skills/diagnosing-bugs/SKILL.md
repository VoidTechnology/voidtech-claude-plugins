# 诊断缺陷

一套针对疑难缺陷的纪律。只有在有明确理由时才跳过某个阶段。

探查代码库时，读 `CONTEXT.md`（若存在），以建立相关模块的清晰心智模型，并查看你所改动区域内的 ADR。

## 阶段 1 — 构建反馈循环

**这就是这个技能的精髓。** 其余一切都是机械操作。如果你拥有一个**紧密**的针对该缺陷的通过/失败信号——一个会因*这个*缺陷而变红的信号——你就一定能找到原因；二分定位、假设检验、插桩都只是在消费它。如果你没有它，盯着代码看再久也救不了你。

在这里投入超出比例的精力。**要积极，要有创造力，拒绝放弃。**

### 构建它的若干途径——大致按此顺序尝试

1. **失败的测试**，建在任何能触达该缺陷的接缝处——单元、集成、e2e。
2. **Curl / HTTP 脚本**，打向一个运行中的开发服务器。
3. **CLI 调用**，喂一份夹具输入，把 stdout 与已知正确的快照对比。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，对 DOM/console/network 做断言。
5. **重放已捕获的轨迹。** 把一个真实的网络请求 / 载荷 / 事件日志存到磁盘；隔离地让它重新走一遍代码路径。
6. **一次性测试台。** 起一个系统的最小子集（一个服务，mock 掉依赖），用单次函数调用动用缺陷代码路径。
7. **属性 / 模糊循环。** 如果缺陷是"有时输出错误"，跑 1000 个随机输入，找出失败模式。
8. **二分定位测试台。** 如果缺陷出现在两个已知状态（commit、数据集、版本）之间，把"在状态 X 启动、检查、重复"自动化，这样就能 `git bisect run`。
9. **差分循环。** 用同一输入分别跑旧版与新版（或两套配置），对比输出。
10. **HITL bash 脚本。** 最后手段。如果必须有人点击，用 `scripts/hitl-loop.template.sh` 驱动*他们*，让循环仍然有结构。捕获的输出回流给你。

构建对的反馈循环，缺陷就修好了 90%。

### 收紧循环

把循环当成一个产品。一旦你有了*一个*循环，就**收紧**它：

- 能更快吗？（缓存初始化、跳过无关的 init、收窄测试范围。）
- 能让信号更锐利吗？（断言具体症状，而不是"没崩"。）
- 能更确定吗？（固定时间、给 RNG 播种、隔离文件系统、冻结网络。）

一个 30 秒的不稳定循环，比没有循环好不了多少；一个 2 秒的确定性循环则是紧密的——是调试的超能力。

### 非确定性缺陷

目标不是干净的复现，而是**更高的复现率**。把触发器循环跑 100 次、并行化、加压、收窄时序窗口、注入 sleep。一个 50% 概率出错的缺陷可调试；1% 则不行——持续提高复现率，直到可调试为止。

### 当你确实无法构建循环时

停下并明确说出来。列出你尝试过什么。向用户索取：(a) 能复现该问题的环境访问权限，(b) 一份已捕获的产物（HAR 文件、日志转储、core dump、带时间戳的屏幕录像），或 (c) 在生产环境加临时插桩的许可。**不要**在没有循环的情况下贸然进入假设阶段。

### 完成判据 — 一个会变红的紧密循环

当循环既**紧密**又**具备变红能力**时，阶段 1 完成：你能说出**一条命令**——一个脚本路径、一次测试调用、一条 curl——它是你**已经至少跑过一次的**（贴出调用方式及其输出），并且满足：

- [ ] **具备变红能力** — 它驱动真实的缺陷代码路径，并断言**用户的确切症状**，因此能因这个缺陷变红、修复后变绿。不是"运行不报错"——它必须能_捕获这个具体缺陷_。
- [ ] **确定** — 每次运行结论相同（不稳定缺陷：按上文，固定一个高复现率）。
- [ ] **快** — 以秒计，而非以分钟计。
- [ ] **可由 agent 运行** — 你能无人值守地跑它；只有通过 `scripts/hitl-loop.template.sh` 才允许人介入循环。

如果你发现自己在这条命令存在之前就开始读代码来构建理论，**停下——直接跳到假设正是这个技能要防止的失败。** 没有具备变红能力的命令，就不进入阶段 2。

## 阶段 2 — 复现 + 最小化

跑循环。看着它变红——缺陷出现。

确认：

- [ ] 循环产生的是**用户**描述的那个失败模式——而不是恰好相邻的另一个失败。错的缺陷 = 错的修复。
- [ ] 该失败在多次运行中可复现（或对非确定性缺陷，复现率高到足以拿来调试）。
- [ ] 你已捕获确切症状（错误信息、错误输出、缓慢时序），好让后续阶段能验证修复确实对症。

### 最小化

一旦变红，把复现收缩到**仍会变红的最小场景**。逐项削减输入、调用方、配置、数据、步骤——**一次削一项**，每削一次都重跑循环——只保留对失败起承重作用的东西。

为何要这么做：最小复现缩小了阶段 3 的假设空间（剩下可疑的活动部件更少），并在阶段 5 成为干净的回归测试。

当**每个剩余元素都起承重作用**时即完成——去掉其中任意一个都会让循环变绿。

在你既复现**又**最小化之前，不要继续。

## 阶段 3 — 假设

在检验任何假设之前，先生成 **3–5 个排好序的假设**。只生成单个假设会让你锚定在第一个看似合理的想法上。

每个假设都必须**可证伪**：陈述它做出的预测。

> 格式："若 <X> 是原因，则 <改变 Y> 会让缺陷消失 / <改变 Z> 会让它更严重。"

如果你说不出预测，那这个假设就是一种感觉——丢弃它，或把它磨锐。

**在检验之前把排序列表给用户看。** 他们常常有领域知识能瞬间重排（"我们刚给 #3 上线了一个改动"），或知道哪些假设已被排除。廉价的检查点，巨大的省时。但别为此卡住——若用户离线，就按你的排序推进。

## 阶段 4 — 插桩

每个探针都必须对应阶段 3 的某个具体预测。**一次只改一个变量。**

工具偏好：

1. **调试器 / REPL 检查**，若环境支持。一个断点胜过十条日志。
2. **定向日志**，打在能区分各假设的边界处。
3. 绝不"打印一切再 grep"。

**给每条调试日志打标签**，比如统一前缀 `[DEBUG-a4f2]`。最后清理就成了一次 grep。没打标签的日志会残留；打了标签的会被清掉。

**性能分支。** 对性能退化，日志通常用错了。改为：先建立基线测量（计时测试台、`performance.now()`、profiler、查询计划），再做二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复**之前**写回归测试——但仅当存在**正确的接缝**时。

正确的接缝是指：测试能动用缺陷在调用点发生时的**真实模式**。如果唯一可用的接缝太浅（缺陷需要多个调用方却只做了单调用方测试、单元测试无法复现触发缺陷的链路），在那里写回归测试只会给你虚假的信心。

**如果不存在正确的接缝，这本身就是一个发现。** 记下它。代码库架构正在阻碍这个缺陷被锁定。把这点标记给下一阶段。

如果存在正确的接缝：

1. 把最小化复现变成该接缝处一个失败的测试。
2. 看着它失败。
3. 施加修复。
4. 看着它通过。
5. 对原始（未最小化的）场景重跑阶段 1 的反馈循环。

## 阶段 6 — 清理 + 复盘

宣布完成前必做：

- [ ] 原始复现不再复现（重跑阶段 1 的循环）
- [ ] 回归测试通过（或已记录接缝缺失）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 该前缀）
- [ ] 一次性原型已删除（或移到明确标注的调试位置）
- [ ] 在 commit / PR 信息中陈述最终被证明正确的那个假设——好让下一个调试者学到东西

**然后自问：什么能在源头防住这个缺陷？** 如果答案涉及架构改动（没有好的测试接缝、调用方纠缠、隐藏耦合），带着具体情况交给 `/improve-codebase-architecture` 技能。在修复落地**之后**再给建议，而非之前——此刻你掌握的信息比开始时多。
