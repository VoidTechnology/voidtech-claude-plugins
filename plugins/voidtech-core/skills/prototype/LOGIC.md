一个微型的交互式终端应用，让用户亲手驱动一个状态模型。当问题关乎**业务逻辑、状态转换或数据形态**时使用它——也就是那种在纸面上看着合理、却只有把它推过真实情形后才察觉不对劲的东西。

## 什么时候这是对的形态

- "我不确定这个状态机能否处理先 X 再 Y 的边界情况。"
- "这个数据模型真的能让我表达……的情况吗？"
- "我想在动手写之前摸清这个 API 应该长什么样。"
- 任何用户想要**按按钮并观察状态变化**的场景。

如果问题是"这东西应该长什么样"——选错分支了。用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在写代码之前，先写下你要给什么状态模型、回答什么问题做原型。一段话，写在原型的 README 里或文件顶部的注释里。回答错问题的逻辑原型纯属浪费——把问题写明确，以便日后核对，无论用户此刻是否在旁观，还是事后才 AFK 回头看。

### 2. 选择语言

用宿主项目所用的语言。如果项目没有明显的运行时（例如一个文档仓库），就询问。

在工具链上沿用项目现有的约定——不要仅仅为了原型就引入一个新的 package manager 或运行时。

### 3. 把逻辑隔离进一个可移植的模块

把真正回答问题的那部分逻辑，藏在一个小巧、纯粹、日后能被整体抽出并放进真实代码库的接口背后。环绕它的 TUI 是用完即弃的；逻辑模块则不应是。

合适的形态取决于问题：

- **一个纯 reducer** —— `(state, action) => state`。当动作是离散事件、状态是单一值时合适。
- **一个状态机** —— 显式的状态与转换。当"此刻到底哪些动作是合法的"本身就是问题的一部分时合适。
- **一小组纯函数**，作用于一个朴素数据类型之上。当没有隐含的当前状态、只有变换时合适。
- **一个有清晰方法接口的 class 或模块**，当逻辑确实拥有持续的内部状态时。

挑选最契合所问问题的形态，*而不是*最容易接到 TUI 上的那个。保持纯粹：不做 I/O，不写终端代码，不为控制流写 `console.log`。TUI 导入它并调用它；不存在反方向的流动。

这正是让原型在自身生命周期之后仍然有用的原因。当问题被回答后，验证过的 reducer / 状态机 / 函数集可以被抽进真实模块——TUI 外壳则被删除。

### 4. 构建能暴露状态的最小 TUI

把它构建成一个**轻量 TUI**——在每一个 tick，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价物）并重渲染整帧。用户应当始终看到一个稳定的视图，而不是不断增长的滚动回看记录。

每一帧有两部分，按此顺序：

1. **当前状态**，漂亮地打印出来且便于 diff（每行一个字段，或格式化的 JSON）。对字段名或区块标题用**加粗**，对次要的上下文（时间戳、ID、派生值）用**暗淡**。原生 ANSI 转义码即可——`\x1b[1m` 加粗、`\x1b[2m` 暗淡、`\x1b[0m` 重置。除非项目里已有，否则无需引入一个样式库。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。把按键加粗、描述暗淡，或反之——怎么读着清爽就怎么来。

行为：

1. **初始化状态** —— 一个单一的内存对象/结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行）**，分派给一个修改状态的 handler。
3. 每次动作后**重渲染**整帧——别追加，要替换。
4. **循环直到退出。**

整帧应当能放进一屏。

### 5. 让它一条命令就能运行

在项目现有的 task runner（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）里加一个脚本。用户应当能运行 `pnpm run <prototype-name>` 或等价命令——绝不需要记住某个路径。

如果宿主项目没有 task runner，就把命令放在原型 README 的顶部。

### 6. 交接

把运行命令交给用户。他们会自己驱动它；最有意思的时刻是他们说"等等，这不该可能发生"或"咦，我本以为 X 会不一样"——这些是*想法*里的 bug，而这正是整件事的意义所在。如果他们想加新动作，就加。原型是会演化的。

### 7. 记录答案

当原型完成了它的使命，对问题的答案是唯一值得保留的东西。如果用户在场，问问它让他们学到了什么。如果不在，就在原型旁留一个 `NOTES.md`，好让答案能被填上（如果你旁观了整场会话，也可以由你来填），然后再删除原型。

## 反模式

- **不要加测试。**一个需要测试的原型就不再是原型了。
- **不要接到真实数据库。**用内存存储，除非问题专门就是关于持久化。
- **不要做泛化。**没有"万一以后我们想支持 X 呢"。原型只回答一个问题。
- **不要把逻辑和 TUI 搅在一起。**如果 reducer / 状态机里引用了 `console.log`、prompt 或终端转义码，它就不再可移植了。把 TUI 当作一层薄壳，套在一个纯模块之上。
- **不要把 TUI 外壳发到生产环境。**外壳是为从终端手动驱动而优化的。它背后的逻辑模块才是值得保留的部分。
