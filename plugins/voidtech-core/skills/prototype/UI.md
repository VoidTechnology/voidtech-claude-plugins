在单个路由上生成**几种截然不同的 UI 变体**，通过一个浮动的底部栏切换。用户在浏览器里来回切换各变体，选定一个（或从每个里各取一点），然后把其余的扔掉。

如果问题关乎逻辑/状态而非外观——选错分支了。用 [LOGIC.md](LOGIC.md)。

## 什么时候这是对的形态

- "这个页面应该长什么样？"
- "我想在拍板之前先看几个这个仪表盘的方案。"
- "给设置页试一个不同的布局。"
- 任何用户本会花一整天在脑子里三个模糊的脑内草图之间纠结的时候。

## 两个子形态——强烈优先子形态 A

当 UI 原型**紧贴着 app 的其余部分**时——真实的 header、真实的 sidebar、真实的数据、真实的密度——它要好判断得多。一个孤立的用完即弃路由是个真空：每个变体单独看都还行。只要有一个合理的现有页面能承载这些变体，就默认选子形态 A。只有当原型确实没有就近的归处时，才去用子形态 B。

### 子形态 A —— 对现有页面的调整（首选）

路由已经存在。变体**渲染在同一路由上**，由一个 `?variant=` URL search param 把关。现有的数据获取、参数和鉴权全部保留——只有渲染部分被替换。这是默认选择；除非有特定理由，否则就选它。

如果原型是为某个尚无页面、但*天然会归属于某个页面内部*的东西做的（仪表盘的一个新区块、设置页上的一张新卡片、某个现有流程里的一个新步骤）——那它仍然是子形态 A。把变体挂载进宿主页面里。

### 子形态 B —— 一个新页面（最后手段）

仅当被做原型的东西确实没有任何现有页面可以归属时才用它——例如一个全新的顶层界面，或一个无处可合理嵌入的流程。

遵循项目已有的路由约定，创建一个**用完即弃的路由**——不要发明新的顶层结构。给它取个一眼能看出是原型的名字（例如在路径或文件名里包含 `prototype` 一词）。用同样的 `?variant=` 模式。

在拍板用子形态 B 之前，先做个常识检查：真的没有任何现有页面能把它嵌进去吗？空路由会掩盖那些有内容的页面本会暴露的设计问题。

两个子形态里，浮动的底部栏完全相同。

## 流程

### 1. 陈述问题并定下 N

默认 **3 个变体**。超过 5 个就不再是截然不同，而是开始变成噪声——封顶在那里。

把计划用一行写下来，写在原型所在处或文件顶部的注释里：

> "设置页的三个变体，通过 `?variant=` 切换，挂在现有的 `/settings` 路由上。"

无论用户在不在场来提反对意见，这都成立。

### 2. 生成截然不同的变体

起草每个变体。让每个都达到：

- 页面的用途，以及它能访问到的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS，随便什么）。
- 一个清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**——不同的布局、不同的信息层级、不同的主要操作方式，而不只是颜色不同。三个略加调整的卡片网格不是 UI 原型，是墙纸。如果两份草稿出来太相似，就用明确的"不要用卡片网格"的指引重做其中一个。

### 3. 把它们接起来

在路由上创建单个切换器组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：把所有现有的数据获取保留在切换器之上；每个变体只改变被渲染的子树。

对于子形态 B（新页面）：`/prototype/<name>` 下的用完即弃路由挂载同一个切换器。

### 4. 构建浮动切换器

屏幕底部居中的一个小小的固定定位栏，含三个部件：

- **左箭头** —— 切到上一个变体（循环回绕）。
- **变体标签** —— 显示当前变体键，如果变体导出了名字，也一并显示。例如 `B — Sidebar layout`。
- **右箭头** —— 向前切换（循环回绕）。

行为：

- 点击箭头更新 URL search param（用框架的 router——Next 上用 `router.replace`，React Router 上用 `navigate` 等），这样变体就可分享、刷新后也稳定。
- 键盘：`←` 和 `→` 方向键也可切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 处于焦点时，不要拦截方向键。
- 在视觉上与页面明显区分（例如高对比度的胶囊形、微妙的阴影），这样一眼就能看出它不是被评估的设计的一部分。
- 在生产构建中隐藏——以 `process.env.NODE_ENV !== 'production'` 或等价检查把关，这样一次走神的原型合并也不会把这个栏发给用户。

把切换器放在一个单一的共享组件里，以便两个子形态都能复用。把它放在项目中共享 UI 所在的地方。

### 5. 交接

把 URL（以及 `?variant=` 键）摆出来。用户有空时会逐个翻看。最有意思的反馈通常是**"我想要 B 的 header 配 C 的 sidebar"**——那才是他们真正想要的设计。

### 6. 记录答案并清理

一旦某个变体胜出，写下是哪一个以及为什么（commit message、ADR、issue，或者在 AFK 运行且用户尚未回应时紧挨着原型放一个 `NOTES.md`）。然后：

- **子形态 A** —— 删掉落选的变体和切换器；把胜出者折叠进现有页面。
- **子形态 B** —— 把胜出的变体提升为真实路由，删掉用完即弃的路由和切换器。

不要把变体组件或切换器丢在那儿。它们烂得很快，会把下一个读代码的人搞糊涂。

## 反模式

- **只在颜色或文案上不同的变体。**那是微调，不是原型。真正的变体在结构上各执一词。
- **变体之间共享太多代码。**共享一个 `<Header>` 没问题；共享一个 `<Layout>` 就背离了初衷。每个变体都应能自由地把布局推翻重来。
- **把变体接到真实的 mutation 上。**只读原型没问题。如果某个变体需要 mutate，就让它指向一个 stub——问题是"这东西应该长什么样"，而不是"后端能不能用"。
- **把原型直接提升到生产环境。**变体代码是在原型约束下写的（没有测试、错误处理极少）。折叠进去时要正经重写一遍。
