# 原型验证

原型是为回答一个明确问题而写的一次性代码。要验证的问题决定原型采用终端程序还是 UI 页面。

## 选择一个分支

确认要回答的是哪个问题——可以从用户的提示、周围的代码中判断，或者在用户在场时直接询问：

- **"这套逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个微型的交互式终端应用，把状态机推过那些在纸面上难以推演的情形。
- **"这个界面应该长什么样？"** → [UI.md](UI.md)。在单个路由上生成几种截然不同的 UI 变体，通过 URL search param 和一个浮动的底部栏来切换。

两个分支产出的产物迥然不同——选错就会浪费掉整个原型。如果问题确实含糊不清且用户又联系不上，那就默认选择与周围代码更匹配的那个分支（后端模块 → 逻辑；页面或组件 → UI），并在原型顶部说明这个假设。

## 两个分支都适用的规则

1. **从创建时就明确标为一次性代码。**把原型放在相关模块或页面附近，让上下文清楚；文件名或目录名必须让人一眼看出它不是生产代码。一次性 UI 路由应沿用项目已有约定，不要新增顶层结构。
2. **一条命令即可运行。**用项目现有 task runner 所支持的任何方式——`pnpm <name>`、`python <path>`、`bun <path>` 等等。用户不应为了启动原型再去查文件路径。
3. **默认不做持久化。**状态存活于内存中。持久化恰恰是原型要*检验*的东西，而不应是它所依赖的东西。如果问题明确涉及数据库，就连到一个临时的 scratch DB，或一个名字明确标着 "PROTOTYPE — wipe me" 的本地文件。
4. **跳过精修。**不写测试，不写超出"让原型能跑起来"所需的错误处理，不做抽象。重点是快速得到结论，然后删除原型。
5. **暴露状态。**每次动作之后（逻辑分支）或每次切换变体时（UI 分支），打印或渲染出全部相关状态，让用户能看到发生了什么变化。
6. **验证结束后清理。**原型回答问题后，要么删除，要么把验证过的决策移入正式代码，不要长期留在仓库中。

## 完成时

原型里唯一值得保留的就是**答案**。把它连同它所回答的问题一起，记录到某个持久的地方（commit message、ADR、issue，或紧挨着原型的 `NOTES.md`）。如果用户在场，这份记录就是一次简短的交谈；如果不在，就留下占位，好让他们（或下一轮的你）在删除原型之前填上结论。
