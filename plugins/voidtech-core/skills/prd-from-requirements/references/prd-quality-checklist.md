# PRD 质量检查清单

## 1. 完整性

- 是否完整读取原始需求。
- 是否覆盖所有系统、模块、子模块、功能点。
- 是否每个模块都有独立文件夹。
- 是否每个模块都有 `prd.md`。
- 是否存在空文件、空目录或只有 TODO 的文档。
- 是否检查目标目录，避免覆盖已有内容。
- 是否按 `prd/YYYYMMDD-{slug}/` 或用户明确指定目录输出。
- 输出目录、系统目录和模块目录 slug 是否使用 English kebab-case，且没有中文、拼音或空格。
- 单系统产品是否避免套用无意义的空系统层，防止系统汇总 PRD 与根汇总 PRD 重复。
- 原始需求文件是否已拷入 `_source/original/`，或已记录校验和；权威来源是否指向工作树内路径。
- 跨端复用对象是否落在 `domain-specs/` 只定义一次；多租户/套餐制产品是否有功能开通矩阵。

## 2. 可追溯性

- 是否每条原始需求都有编号，且编号零填充格式全树一致。
- 是否每条需求能追溯到输出文件和章节。
- 是否标注部分覆盖、待确认、不适用。
- 是否区分原文需求、推断和推荐默认方案，且统一使用 `[推断]`、`[推荐默认]` 方括号标记。
- 期次是否以追溯矩阵为唯一权威，模块文档引用与矩阵无冲突。
- 骨架级的区间级追溯是否做了覆盖率对账（输入总数 = 映射总数）。
- 交付物是否自足：没有用「见 REQ-xxx」回指原始需求代替写出内容。

## 3. 可研发拆解

- 模块边界是否清晰。
- 批量生成 PRD 工作树前，是否已向用户确认模块初分、跨系统依赖和高风险开放问题。
- 功能入口、前置条件、系统响应是否明确。
- 字段、状态、权限、异常是否足够支撑任务拆分。
- 跨系统依赖是否显式。

## 4. 可测试验收

- 验收标准是否可判真伪。
- 是否覆盖正常路径、异常路径、权限路径、边界路径。
- 交互层边缘状态（加载中、空数据、超长数据、弱网、会话过期、权限被拒）是否由端 PRD 逐一定义，而不是指望领域规格兜底。
- 对不适用的验收维度，是否明确标注“不适用”及原因，而不是编造不存在的能力。
- 是否避免“支持某功能”这类不可验收表述。
- 验收级文档是否不含「样例」字样；验收标准中是否没有内嵌未决规则（「按约定规则」「是否即时生效待确认」）。

## 5. 可上线判断

- 是否有本期包含和本期不做。
- 是否有开放问题排序。
- 是否识别阻塞研发、阻塞上线或影响架构的问题。
- 是否给出评审顺序。
- 成功指标是否包含反指标，避免指标被错误优化。

## 6. 风险处理

- 原文含糊处是否进入开放问题。
- 原文矛盾处是否保留并说明影响。
- 是否避免私自抹平退款、支付、权限、安全、外部系统打通等高风险问题。
- 被引用的 OQ 编号是否都有定义行；「来源：开放问题」是否都带编号；是否没有把新问题骑在已关闭的 OQ 编号上。
- 是否不存在两份文档互相指认对方持有、实际两边都没有的信息（循环引用）。
- 评审返回的缺陷是否逐条处置（修复/转排期/转开放问题）并在最终回复对账，无静默丢弃。

## 6a. 跨文档一致性

- 正文引用的状态是否都在对应对象的状态机中定义（无幽灵状态）。
- 同一业务对象的终态归宿是否只在一处裁决，引用方无冲突口径。
- 「见 §N」「依 XX 文档」的指向处是否真实存在内容（无空指针）。
- 需求编号零填充是否全树一致。
- 每个深化 pass 收尾是否回扫了术语表、跨系统依赖、开放问题、功能开通矩阵，全局文档未落后于模块深化。
- 变更记录宣称完成的事项是否逐条属实。

## 7. 深度与分期

- 每份文档头部是否声明深度（骨架级/验收级）。
- 需求规模超过阈值时，是否在确认检查点提出分期计划并得到用户确认。
- 分期交付是否生成 `deepening-backlog.md`，且清单与各文档实际深度一致。
- 最终回复是否如实声明当前深度和尚未适用的红线，未以骨架级冒充验收级。

## 8. 模板清理与机械自检

- 是否运行 `scripts/check-prd-tree.py` 且错误清零；警告是否在最终回复逐条说明。
- 是否运行 `scripts/generate-dashboard.py` 重新生成状态看板，且看板与本轮深化后的实际状态一致；看板未被手改。
- 是否检查输出中没有残留 `{占位符}`。
- 是否检查文件名和目录名没有残留 `{占位符}`。
- 是否检查输出中没有断链链接；汇总拼接时相对路径是否已重写。
- 是否检查模块 `prd.md` 是唯一主本。
- 是否检查汇总 PRD 标注为生成物，且模块主本修订后已重新生成。
- 重大改动是否追加变更记录。
