# product-manager agent 协同协议

## 协同目标

优先委派 `@voidtech-core:product-manager` 提升 PRD 的产品判断质量，而不是把它当成文案生成器。

不要依赖本机绝对路径定位 agent。插件安装位置可能变化，协同对象应通过插件命名空间或运行时可用的 subagent 名称解析。

如果 `@voidtech-core:product-manager` 不可用，主 Agent 按本文档检查项自行降级评审，并在最终回复中说明降级情况。

## 协同上下文连续性

两次协同必须共享决策上下文。若运行环境支持继续同一个 `@voidtech-core:product-manager` subagent，第二次评审优先续用第一次协同的 subagent 上下文。

如果无法续用同一个 subagent，主 Agent 必须把第一次协同的结论整理为“第一次协同决策摘要”，并作为第二次协同输入。摘要至少包含：

- 已采纳的系统与模块边界。
- 被否决的模块边界及否决理由。
- MVP 范围和本期不做。
- 已确认或待确认的跨系统依赖。
- 高风险开放问题排序。
- 已采用的推荐默认方案。
- 用户在确认检查点中做出的调整。

第二次评审不得无理由推翻第一次已采纳且已被用户确认的边界决策；如确需推翻，必须明确说明新证据、影响范围和建议处理方式。

## 主 Agent 负责

- 完整读取原始需求。
- 提取需求点。
- 建立追溯矩阵。
- 生成文件结构。
- 写入 PRD 文档。
- 进行链接、覆盖率、验收标准、状态机、权限矩阵自检。

## product-manager agent 负责

- 自行读取原始需求文件或原始需求正文入口。
- 产品定位审查。
- 产品模块边界审查。
- 用户场景边界审查。
- 用户角色与场景审查。
- 跨系统依赖识别。
- 原文矛盾识别。
- MVP 范围和本期不做判断。
- 高风险开放问题排序。
- 推荐默认方案。
- PRD 完稿前评审。
- 只返回判断和建议，不写入、修改或删除任何文件。

## 第一次协同输入

主 Agent 应向 product-manager agent 提供：

- 原始需求文件路径，或原始需求正文入口；如需求来自多个文件，提供主需求文件和直接引用附件路径。
- 主 Agent 的系统和模块初分草案。
- 主 Agent 的需求点清单草案。
- 主 Agent 已识别的跨系统依赖草案。
- 主 Agent 已识别的开放问题草案。
- 明确约束：只返回判断，不写文件。

请求 product-manager agent 输出：

- 模块边界调整建议。
- 关键业务链路。
- 高风险依赖。
- MVP 范围建议。
- 开放问题排序。

## 第二次协同输入

主 Agent 应向 product-manager agent 提供：

- 原始需求文件路径，或原始需求正文入口。
- 第一次协同决策摘要；若续用了同一个 subagent，也要提供摘要供对照。
- 已生成的 PRD 工作树摘要。
- 覆盖率摘要。
- 高风险开放问题。
- 代表性模块文档。
- 明确约束：只返回判断，不写文件。

请求 product-manager agent 输出：

- 是否能支撑产品评审。
- 是否能支撑研发拆解。
- 是否能支撑测试验收。
- 缺失的关键决策。
- 建议修订项。
- 是否发现必须推翻第一次协同结论的新证据；如有，列出证据和影响范围。

## 重要边界

- product-manager agent 的判断是产品分析，不是原始需求。
- 如果与原始需求不一致，必须保留冲突并进入开放问题。
- 推荐默认方案不能写成已确认结论。
- product-manager agent 虽然具备 Write 权限，但在本协同协议中不得写入、修改或删除文件。
- 如果 product-manager agent 需要落盘文档，必须回到主 Agent，由主 Agent 按用户确认后的输出策略执行。
- product-manager agent 只负责产品模块边界、用户场景边界、业务规则边界。
- 技术模块边界、API 边界、数据库模型、系统架构、部署方案不应由 product-manager agent 定案。
- 如 PRD 需要技术架构判断，应交给 architect 类 agent 或列入开放问题。
