# prd-from-requirements

本技能用于从原始需求生成高质量 PRD 工作树。

## 与 to-prd 的分工

本技能适用于“从原始需求文档生成模块化 PRD 工作树”，主要服务专业产品经理、业务分析、研发评审和测试验收。

如果用户只是希望把当前对话整理成一份轻量单体 PRD，并发布到项目 issue 追踪器，应该改用 `voidtech-core:to-prd`。`to-prd` 不重新访谈需求，也不产出模块化工作树。

对已生成工作树的后续维护——深化模块到验收级、合入需求变更、定案开放问题、落实评审修订——改用 `voidtech-core:prd-maintain`，不要重跑本技能。`prd-maintain` 复用本技能的红线、模板与脚本，只封装维护流程。源需求文件（如 Excel）后续出新版本时的同步走 `voidtech-core:prd-sync`：它只计算差异并提出候选，确认后的合入仍由 `prd-maintain` 执行。本技能只负责首建。

## 触发场景

当用户要求基于原始需求、Excel 整理稿、访谈纪要、需求清单或已有粗稿生成、重构、评审 PRD 时，使用本技能。

典型请求包括：

- 根据原始需求生成 PRD
- 把需求整理成产品需求文档
- 生成模块化 PRD 工作树
- 生成需求追溯矩阵
- 检查 PRD 是否能支撑研发和测试
- 将业务需求拆成可评审、可开发、可验收的文档

## 目标

把原始需求转化为高质量 PRD 工作树，而不是简单改写原文。

产出必须做到：

- 需求可追溯
- 模块边界清晰
- 跨系统依赖显式
- 核心流程可执行
- 状态机可判断
- 字段规则可落地
- 权限矩阵可测试
- 验收标准可判真伪
- 开放问题不被私自定案

## 输入

用户至少需要提供：

- 原始需求路径或原始需求正文
- 输出目录（可选；未提供时按“输出目录安全策略”生成默认目录）

如果需求来源包含多个文件，优先完整读取主需求文件，再读取直接引用的附件、表格、旧版 PRD 或补充说明。

## Excel 输入处理

如果原始需求是 `.xlsx`，不要直接用 Read 读取二进制文件。目标是先得到可审阅的文本中间产物，再读取 Markdown 或 CSV。按下面的降级阶梯处理：

1. **优先使用技能自带脚本**。该脚本只使用 Python 标准库，不依赖 pandas、openpyxl 等第三方包。

   ```bash
   python3 "${CLAUDE_PLUGIN_ROOT}/skills/prd-from-requirements/scripts/xlsx-to-markdown.py" \
     <原始需求.xlsx> \
     <输出目录>/_source/<需求文件名>/
   ```

2. **如果没有 `python3`，探测本机常见转换工具**。优先保留多 sheet 信息。

   ```bash
   command -v xlsx2csv
   command -v in2csv
   ```

   - 有 `xlsx2csv` 时，使用 `xlsx2csv -a <原始需求.xlsx> <输出目录>/_source/<需求文件名>/` 导出所有 sheet。
   - 有 `in2csv` 时，先用 `in2csv -n <原始需求.xlsx>` 查看 sheet，再按 sheet 导出到 `_source/<需求文件名>/`。

3. **如果以上工具都不可用，停下请求用户转换**。明确告知用户：请在 Excel、WPS 或 Numbers 中把需求表另存为 `.csv` 后重新提供；拿到 CSV 前不要假装已经读取 `.xlsx` 内容。

转换产物固定放在当前 PRD 输出目录的 `_source/<需求文件名>/` 下，并标注为“转换产物，以原始 xlsx 为准”：

- `INDEX.md`：sheet 清单和转换结果索引。
- `NN-{sheet}.md`：每个 sheet 的 Markdown 表格，优先供模型阅读。
- `NN-{sheet}.csv`：每个 sheet 的 CSV，供精确检索、计数或复核。

注意：

- 日期单元格可能显示为 Excel 序列数字。遇到疑似日期的五位数字列，例如上线时间、截止日期、生效日期，必须先与用户确认口径，不要直接照抄进 PRD。
- 技能自带脚本会按 `.xlsx` 中的合并单元格定义回填合并区域；如果使用降级工具后仍出现分类列、模块列、系统列的大量空白单元格，先对照 CSV、上方单元格和原始表格判断归属，不要直接判定为需求缺失。

读取顺序：

1. 先读 `_source/<需求文件名>/INDEX.md`，确认 sheet 列表。
2. 再逐个读取相关 `NN-{sheet}.md`；必要时读取对应 CSV 复核字段、行数和原始单元格内容。
3. 在 PRD 的“原始需求来源”中引用原始 `.xlsx` 路径和转换产物路径，但追溯矩阵的权威来源仍写原始 `.xlsx`，转换产物只作为读取辅助。

如果输入是旧版 `.xls`，脚本无法直接解析。先要求用户另存为 `.xlsx` 或 `.csv`；不要假装已经读取了二进制 `.xls`。

## 权威源管理

追溯的前提是权威源不丢。规则：

- 生成前把原始需求文件拷贝进 `<输出目录>/_source/original/`；追溯矩阵和各文档的「权威来源」一律指向工作树内路径。
- 无法拷贝时（文件过大、用户拒绝），必须在追溯矩阵记录原路径、文件大小和 SHA-256 校验和（`shasum -a 256 <文件>`），并注明「原件不在工作树内」。
- 禁止把工作树外的本机绝对路径作为唯一权威来源：原件一旦移动或删除，「以原始文件为准」「落盘前对照原件确认」都会变成不可执行的指令。

## 输出目录安全策略

生成或修改文件前，必须先检查目标目录是否存在。

默认输出目录规则：

- 若用户明确指定输出目录，使用用户指定目录。
- 若用户未指定输出目录，默认创建 `prd/YYYYMMDD-{slug}/`，其中 `{slug}` 来自产品名或需求文件名。
- 不要默认直接写入已有 `prd/` 根目录。
- 输出目录、系统目录和模块目录中的 slug 必须使用 English kebab-case，例如 `order-center`、`member-level`；不要使用中文、拼音或空格。

已有内容处理规则：

- 目标目录不存在：可直接创建。
- 目标目录存在且为空：可直接写入。
- 目标目录存在且已有文件：不得覆盖、删除或重写已有内容，除非用户明确确认。
- 如果需要在已有目录上增量更新，必须先说明将新增、修改、保留哪些文件。
- 如果用户要求重建，优先将旧内容保留到 `archive/YYYYMMDD-HHMMSS/`，再生成新版本；不要静默覆盖。

强制路由检查点：若目标目录已经是 PRD 工作树（存在 `00-global/requirement-traceability-matrix.md` 或模块 `prd.md`），且用户意图是更新、深化、修订或「用新版技能更新」，必须停下向用户三选一，确认前不得动手：

1. 改用 `voidtech-core:prd-maintain`（推荐——它有工况约束和收尾不变式，本技能没有「更新」剧本）。
2. 全量重建：旧版先归档到 `archive/YYYYMMDD-HHMMSS/`。
3. 按明确的增量清单执行：先列出将改哪些文件、对应哪种维护工况，经确认后照单执行。

一句「更新 PRD」不构成增量授权。实测教训：没有工况约束的增量更新，模型会只做机械检查能看见的便宜修复，跳过内容深化，再把进度表标成已完成。

## 深度分级与分期交付

模块 PRD 有两档交付深度。模块与系统总览 `prd.md`、领域规格、功能开通矩阵、追溯矩阵和两级汇总 PRD 必须在头部声明自己处于哪一档（术语表、跨系统依赖等全局文档不要求）：

- **骨架级**：结构、模块边界、需求区间级追溯、领域规格中的状态机与词表裁决、开放问题、样例验收。用于确认范围和边界，不可进开发排期。
- **验收级**：满足 `templates/module-prd.md` 全部章节——逐条需求的功能清单、页面契约、核心流程、流程状态影响、页面交互、页面↔数据读写、完整状态机/权威引用、边缘状态、字段规则（类型/长度/校验）、角色权限矩阵、编号验收标准、行级追溯。只有验收级模块可以进入开发排期。

分期判断（作为用户确认检查点的一部分提交）：

- 需求点少于约 100 条且模块不超过 8 个：默认单次交付验收级。
- 超过该规模：一次会话物理上写不完验收级全量，不得假装写完。必须提出分期计划——第一期交付骨架级工作树和全部全局文档，之后按模块逐个深化到验收级——并给出建议深化顺序。
- 分期交付时必须生成 `00-global/deepening-backlog.md`（模板见 `templates/deepening-backlog.md`），作为深化进度的唯一主本。每深化一个模块，同步更新该模块头部深度声明、追溯矩阵粒度和这份清单。
- 首次交付之后的深化 pass 属于维护，由 `voidtech-core:prd-maintain` 承接（它按本技能的规则与模板执行），不需要重跑本技能。

红线按深度档适用：

- 骨架级交付不豁免红线，只是延后验收类红线的适用时点。最终回复必须写明当前深度、哪些红线尚未适用、深化顺序，不得宣称已完成。
- 验收级模块必须满足全部质量红线；验收标准不允许出现「样例」字样。
- 验收级不是「章节填满」：升级前必须过一遍深化任务清单模板里的「跨文档一致性自检」（幽灵状态、终态唯一、无循环互指、编号格式统一）。实测中虚标验收级的主要方式不是缺章节，而是填了表但表间互相矛盾。
- 验收级由评审写入，不允许自我认证。生成或深化完成后，backlog 对应行先标「待评审」，文档头部暂不标「验收级」；由 `@voidtech-core:product-manager` 按 DoD 与跨文档一致性自检逐项核验并留证据（grep 结果、对照行号），记录写入 backlog 的「验收级核验记录」表，核验通过才可改头部深度和「已完成」状态。subagent 不可用时主 Agent 降级自审，必须在记录中注明降级并同样逐项留证。机械自检会校验：标「验收级」的文档必须在核验记录表有对应条目。实测教训：两轮里自检被当成打勾动作，做工的 agent 给自己的 7 行全标了「无缺口」，独立复审推翻了其中 6 行。
- 验收级模块的审计结构是 Definition of Done，不是可选的 Logic Atlas 增强项。用户界面模块必须用固定表完整声明「页面 → 流程步骤 → 页面交互 → 状态变化/异常/依赖 → 数据读写」；纯服务模块也必须保留对应章节并写明「不涉及」及原因。空表、只写自然语言、只补少数样例，均不得通过验收级评审。
- 评审选择试点模块时，优先选同时跨页面、状态、权限和外部依赖的主流程模块；试点必须把上述审计结构补全并发布真实 Atlas 验证。不得用编译器 fixture、只读场景或边界模块替代真实业务闭环。

每个深化 pass 收尾时必须回扫全局文档，不允许全局层落后于模块深化：

- 术语表：新出现的多义术语（同名不同义）必须消歧成独立条目。
- 跨系统依赖：模块深化中确认的外部依赖（支付通道、消息服务商等）登记进依赖清单。
- 开放问题：新增推断与默认值补 OQ 编号；已确认的问题更新状态，不得把新问题骑在已关闭编号上。
- 功能开通矩阵：模块引用到的功能点在矩阵中落行。
- 行级追溯的标准做法：深化后的模块在其 `prd.md` 第 13 节逐条成行（含此前用区间引用的部分），追溯矩阵保留区间入口并标注「已深化，行级见模块第 13 节」；矩阵头部状态措辞同步更新，不得残留「将在……落地」这类过时表述。深化任务清单与追溯矩阵对「行级」的定义必须一致。

## 评审缺陷处置

协同评审（product-manager agent 评审或用户评审）返回的每一条缺陷，只允许三种处置，且必须留痕：

1. **立即修复**：改模块主本并重新生成汇总。
2. **转排期**：登记进 `deepening-backlog.md` 对应行的缺口列，写明缺陷内容。
3. **转开放问题**：立 OQ 编号进全局清单。

禁止静默丢弃。最终回复必须附「缺陷处置清单」：修复 N 条、转排期 N 条（列条目）、转开放问题 N 条（列编号），三类数量之和必须等于评审返回的缺陷总数。

## 领域规格层

同一业务对象被两个及以上端或系统引用时（多端产品几乎必然如此），必须建立 `00-global/domain-specs/` 领域规格层：

- 跨端复用的对象、状态机、字段规则、词表裁决、业务逆向边缘状态，在领域规格中只定义一次（模板见 `templates/domain-spec.md`）。
- 端/模块 PRD 只写本端可见范围、操作差异和交互层边缘状态（加载中、空数据、弱网、会话过期），并链接领域规格，不复制其内容。
- 原始需求在不同端对同一对象使用不同状态词表时，必须在领域规格中做显式裁决并注明被废弃词表的来源编号，各端不得另建状态。
- 状态一致性两条铁律：正文引用的每个状态必须存在于该对象的状态机中，不得在流程描述里引入未定义状态（幽灵状态）；同一业务对象的终态归宿只在一个领域规格里裁决一次，引用方（其他领域规格、模块 PRD、源需求冲突处）与裁决不一致时改引用方或立开放问题，不允许多头共存。
- 供 Logic Atlas 生成统一场景流程时，模块必须用固定表显式声明「流程步骤 → 页面交互 → 状态变化/异常/依赖 → 页面数据读写」：每条页面交互引用既有流程与步骤，本模块页面引用页面契约，跨模块页面使用 `<module-scope>::<页面名>`；每条页面数据读写同时引用流程、步骤、页面和已声明数据对象；同一步骤的成功链必须唯一入口且可到达「结束」，状态影响与边缘状态通过交互 ID 挂到具体操作。无法验证的关联进入 gaps，不允许按文案相似度猜测，也不得把模块级读写扇形下推到所有页面。
- 多租户、多版本或套餐制产品必须另建 `00-global/feature-gating-matrix.md`（模板见 `templates/feature-gating-matrix.md`），文件名固定不得改名（自检脚本按 `*-matrix.md` 角色检查深度声明，但固定名是引用方回指的契约）；它管功能开通，不能替代模块 PRD 里的角色 × 操作权限矩阵，两边也不允许互相指认对方持有而两边都没有的信息。矩阵本身若为骨架，必须在深化任务清单中登记补全任务。

## 协同 Agent

优先委派 `@voidtech-core:product-manager` 做产品判断。

不要在技能中依赖本机绝对路径定位 agent。插件可能被安装到不同机器、不同用户目录或缓存目录中。

如果 `@voidtech-core:product-manager` 不可用，主 Agent 必须按 `references/pm-agent-collaboration.md` 中的检查项自行降级评审，并在最终回复中说明已降级为主 Agent 自审。

协同分工：

- 主 Agent：读取原始需求、建立文件结构、生成文档、维护追溯矩阵、执行自检。
- product-manager agent：自行读取原始需求文件或原始需求正文，并只返回产品判断，包括模块边界、业务目标、MVP 范围、跨系统依赖、风险排序、开放问题推荐方案。

注意：

- 协同时必须传递原始需求文件路径或原始需求正文入口，以及主 Agent 的草案；不要只传二手摘要。
- 如果运行环境支持继续同一个 `@voidtech-core:product-manager` subagent，上下两次协同优先续用同一个 subagent 上下文。
- product-manager agent 在本技能中只返回判断和建议，不写入、修改或删除任何文件。
- product-manager agent 的判断不能直接当作原始需求。
- 凡非原文内容，必须按「推断标记规范」标注。
- 原文含糊或矛盾时，必须进入开放问题，不允许私自抹平。

## 推断标记规范

凡非原文内容，只允许两种标记，统一写作方括号形式以便机器审计：

- `[推断]`：从原文能合理推出的结论。
- `[推荐默认]`：原文未定、按推荐默认方案先行落笔的内容。必须有对应的全局开放问题编号（OQ-xxx）并在落笔处回指；不允许自定默认值而不进开放问题。

不要用「派生」「默认」「推荐，待确认」等变体代替标记：自检脚本按 `[推断]` 和 `[推荐默认]` 审计，变体会漏检。开放问题一律使用 `OQ-` 前缀编号；文档中写「来源：开放问题」必须带编号，保证能与全局清单对账。

## 用户确认检查点

模块划分、跨系统依赖和高风险开放问题是全局性决策。批量生成 PRD 工作树前，必须先向用户提交确认摘要，并暂停等待明确确认。

确认摘要必须包含：

- 系统与模块初分：系统、模块、子模块、每个模块的一句话职责。
- 跨系统依赖草案：上游、下游、依赖内容、未确认项。
- 高风险开放问题：阻塞研发、影响架构、影响数据模型、影响权限安全、影响上线验收的问题。
- 交付深度与分期计划：本期交付骨架级还是验收级；分期时附建议深化顺序（见「深度分级与分期交付」）。
- 变更影响说明：如果用户调整模块边界，哪些文档结构、追溯矩阵和汇总 PRD 会随之变化。

未得到用户明确确认前，不得批量创建模块目录、模块 PRD、系统汇总 PRD 或全量汇总 PRD。允许在确认前生成一份临时确认摘要，但不得把它当作最终 PRD 工作树。

## 推荐执行流程

1. 完整读取原始需求，不能只读局部章节；把原始需求文件拷入 `_source/original/`（见「权威源管理」）。
2. 提取系统、模块、子模块、功能点。
3. 建立需求点编号与初版追溯矩阵。
4. 生成模块初分与跨系统依赖草案；判断交付深度与分期计划（见「深度分级与分期交付」）。
5. 协同 product-manager agent 审查模块边界、业务目标、MVP 范围、开放问题和高风险依赖。
6. 记录第一次协同决策摘要，包括采纳的模块边界、被否决的模块边界、MVP 范围、跨系统依赖、高风险开放问题排序和推荐默认方案。
7. 向用户提交确认摘要，包含模块初分、跨系统依赖草案、高风险开放问题、交付深度与分期计划和变更影响；暂停等待明确确认。
8. 用户确认后，按模板和确认的深度生成 PRD 工作树；多端复用对象先落领域规格，再写模块 PRD；分期交付时同步生成 `deepening-backlog.md`。
9. 生成全局文档、系统汇总 PRD 和全量汇总 PRD。模块 `prd.md` 是唯一主本；两级汇总 PRD 是生成物，必须在文件头部标注“修改请改模块 PRD 后重新生成”；拼接正文时按「机械自检」中的链接重写规则处理相对路径。
10. 协同 product-manager agent 做一次 PRD 评审；必须把第一次协同决策摘要作为输入，要求评审不得无理由推翻已采纳且已确认的边界决策。
11. 修订模块 PRD、全局文档和追溯矩阵。不得直接把汇总 PRD 当作主本修改。
12. 修订后重新生成系统汇总 PRD 和全量汇总 PRD，避免汇总副本与模块主本冲突。
13. 人工复核追溯覆盖率、验收标准、状态机、权限矩阵、字段规则与深度声明是否一致。
14. 运行机械自检脚本并修复全部错误（见「机械自检」）。
15. 生成状态看板（见「状态看板」）。
16. 使用 `voidtech-core:text-naturalizer` 润色最终文档正文段落，去掉模板腔、抽象话和 AI 写作痕迹。表格、矩阵、状态枚举、字段定义、权限规则、验收标准、编号、标题层级、文件链接、代码块和推断标记不做自然化改写。
17. 输出最终交付摘要。

## 推荐输出结构

默认按以下结构生成。若用户指定其他结构，以用户指定为准，但仍应保持“每个模块一个独立文件夹”。

```text
prd/
├── README.md
├── _source/
│   ├── original/                    # 原始需求文件副本（权威源）
│   └── {需求文件名}/                 # xlsx 等格式的转换产物
├── 00-global/
│   ├── product-overview.md
│   ├── glossary.md
│   ├── cross-system-dependencies.md
│   ├── cross-system-flows.md
│   ├── global-open-questions.md
│   ├── requirement-traceability-matrix.md
│   ├── feature-gating-matrix.md     # 多租户/多版本/套餐制产品必须
│   ├── deepening-backlog.md         # 分期交付时必须
│   ├── status-dashboard.md          # 生成物：状态看板（脚本生成，勿手改）
│   ├── status-dashboard.html        # 同数据可视版
│   └── domain-specs/                # 对象被 2 个以上端引用时必须
│       ├── README.md
│       └── {domain-slug}.md
├── 01-{system-slug}/
│   ├── README.md
│   ├── 00-overview/
│   │   └── prd.md
│   ├── 01-{module-slug}/
│   │   └── prd.md
│   └── {system-slug}-full-prd.md
└── full-prd.md
```

要求：

- 每个系统一个目录。
- 每个模块一个独立文件夹。
- 每个模块文件夹至少包含 `prd.md`。
- 系统和模块 slug 必须使用 English kebab-case，不得使用中文、拼音或空格。
- 复杂模块可额外增加 `state-machine.md`、`data-fields.md`、`permission-matrix.md`、`acceptance.md`。
- 不允许生成空目录、空文件或只有 TODO 的模块文档。
- 不允许生成断链链接。
- 模块 `prd.md` 是唯一主本；`{system-slug}-full-prd.md` 和 `full-prd.md` 是生成物，不能作为后续人工修订入口。
- 单系统产品允许省略 `01-{system-slug}/` 系统层，直接在根目录下放置模块目录和 `full-prd.md`，避免 `{system-slug}-full-prd.md` 与根 `full-prd.md` 完全重复。

## 质量红线

不允许：

- 遗漏原始需求点。
- 只写目录，不写模块内容。
- 用“支持某功能”代替可验收描述。
- 把开放问题私自定案。
- 把推断内容伪装成原始需求。
- 只写正常流程。
- 涉及状态流转却缺少状态机；状态流转缺少触发条件或触发方式。
- 涉及角色差异却缺少权限矩阵。
- 涉及业务对象却缺少字段规则。
- 缺少本期不做。
- 缺少需求追溯。
- 生成断链文件链接。
- 同时维护多个互相冲突的主本；若存在汇总 PRD，必须标注为生成物并在模块主本修订后重新生成。
- 最终输出中残留 `{占位符}`。
- 「深度分级与分期交付」要求声明深度的文档缺少深度声明，或以骨架级产出冒充验收级交付；验收级文档中的验收标准标注「样例」。
- 未经独立核验（核验记录表无对应「通过」条目）就把文档标为验收级、把深化任务标为已完成。
- 在已有 PRD 工作树上，未经「转 prd-maintain / 全量重建 / 明确增量清单」三选一确认就开始增量修改。
- 模块 PRD 或领域规格另设与追溯矩阵冲突的期次口径。期次（MVP/后置/降级/二期）以追溯矩阵为唯一权威；模块文档引用期次必须与矩阵一致，改期次先改矩阵再改引用方。
- 写「来源：开放问题」却没有对应的 OQ 编号和全局清单条目。
- 两份文档互相指认对方持有某信息而两边都没有（循环引用）。
- 正文引入状态机中不存在的状态（幽灵状态），或同一对象的终态归宿在多份文档里各说各话。
- 静默丢弃评审缺陷：评审返回的缺陷既未修复、也未登记深化任务清单或开放问题就消失。
- 用工作树外的本机绝对路径作为唯一权威来源。
- 交付物中回指原始需求编号代替写出内容（如「操作见 REQ-xxx」）；读者不翻原始文件就无法开发的 PRD 不是交付物。

## 机械自检

交付前必须运行技能自带自检脚本，对输出目录做机器检查：

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/prd-from-requirements/scripts/check-prd-tree.py" <输出目录>
```

退出码：0 干净；1 存在检查错误；2 用法错误；3 读取栅栏——已迁移工作树（存在 `prd-worktree.json`）中有未完成的同步 operation，脚本零写入、只报告 operation id。遇 3 先恢复再重跑，不得绕过：

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/prd-from-requirements/scripts/prd-sync.py" recover <输出目录>
```

operation 进行中要自检暂存修改时，加 `--operation-id <id>`，对「当前有效输入 + 该 operation 暂存修改」的预提交视图执行同样检查。legacy 工作树（无 `prd-worktree.json`）不受以上两条影响。

脚本报「错误」的项必须修复后重跑，直到退出码为 0；报「警告」的项不阻塞交付，但必须在最终回复中逐条说明处理方式。检查项：

- 模板占位符 `{...}` 残留：正文（围栏代码块与行内代码不计）以及文件名、目录名。
- 相对路径断链：正文引用的 `.md` 路径必须能从所在文件位置解析到。
- 工作树外绝对路径引用（见「权威源管理」）。
- 汇总 PRD 头部缺「生成物」声明。
- 空文件与 TODO/TBD/FIXME 残留。
- 推断标记审计：`[推断]`、`[推荐默认]` 之外的裸用法，以及「派生」「(默认…)」「推荐/默认…待确认」等变体。
- 开放问题登记一致性：被引用的 OQ 编号必须有定义行；「来源：开放问题」必须带编号；「开放问题 #n」式回指提示改用 OQ- 编号。
- 核心文档头部深度声明缺失（按文档角色匹配：模块与系统总览 `prd.md`、`domain-specs/`、所有 `*-matrix.md`、两级汇总；改名不豁免。报错误）。
- 验收级核验记录：头部标「验收级」的文档必须在 `deepening-backlog.md` 的「验收级核验记录」表中有条目（报错误）。
- 需求编号零填充一致性：同一前缀混用 `PTL-006` 与 `PTL-6` 会让 grep 反查漏检。
- 幽灵状态（启发式）：含状态机的文件里，正文「从 X 回到/转/进入 Y」式引用的状态必须在状态机中定义。文件级判断，同名状态跨对象混用识别不了，终态多头、跨文档矛盾也查不出——这两类靠「跨文档一致性自检」人工核，脚本不能替代。

汇总链接重写规则：把模块 `prd.md` 正文拼进系统汇总或根汇总时，正文里的相对路径必须重写为从汇总文件位置可解析的路径；不重写就是断链，脚本会拦下。

如果 `python3` 不可用，按上述清单用 `rg` 逐项手工检查，并在最终回复中说明已降级为手工自检。如果业务正文确实需要保留大括号示例，必须在最终回复中说明原因。

## 状态看板

机械自检通过后，生成状态看板，让维护者一眼看清「谁能交开发」：

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/prd-from-requirements/scripts/generate-dashboard.py" <输出目录>
```

产出 `00-global/status-dashboard.md`（权威版，进版本管理、可 diff）和同数据的 `status-dashboard.html`（自包含，浏览器直接打开）。规则：

- 看板是**生成物**，从各文档头部深度声明、「引用领域规格」行、追溯矩阵的「ID 区间 → 归属模块」映射、`cross-system-flows.md` 的流程需求编号和机械自检结果自动汇总。禁止手工维护看板内容——手工维护的状态文件必然过时并说谎。
- 就绪判定算依赖闭包：**可交开发 = 自身验收级 + 引用的领域规格全部验收级 + 无机械错误与可疑信号**。模块自身达标但依赖是骨架级时显示「被依赖阻塞」；深度声明达标但机械信号与之矛盾时显示「存疑」——「自报深度」与「机械信号」分列展示，绿灯配信号就是可疑绿灯。
- 端到端路径视图从跨系统流程推导：一条路径就绪要求涉及的全部模块可交开发，否则列出最短板。
- 每次深化 pass 收尾、评审修订后、汇总重新生成时，必须同步重新生成看板。
- 看板的判定依赖头部字段可解析：深度声明用「- 深度：」列表行，「- 引用领域规格：」列出 slug 或 domain-specs 路径。
- 看板识别不了「挂 OQ 占位冒充默认值」这类内容级虚标——「关联 OQ」列供人工抽查，跨文档一致性自检不可省。

## 正文润色

最终交付前使用 `voidtech-core:text-naturalizer` 的规则润色文档正文段落，降低模板腔、抽象话和 AI 写作痕迹。若 Skill 工具可用，调用 `voidtech-core:text-naturalizer` 处理草稿；若不可用，读取 `${CLAUDE_PLUGIN_ROOT}/skills/text-naturalizer/SKILL.md`，按其中规则自审并改写。

润色范围：

- 可润色：产品总览、模块背景、用户目标、业务说明、流程说明、补充说明、开放问题背景等自然语言正文段落。
- 不润色：需求追溯矩阵、跨系统依赖表、权限矩阵、字段表、状态表、验收标准表、开放问题清单、编号、标题层级、文件链接、代码块、原始需求引用。

润色时必须保留事实、范围、术语、需求编号、模块边界、状态名、字段名、角色名、验收条件、开放问题和推荐默认方案，不得为了自然表达而弱化约束或改写结构化内容。

## 最终回复

完成后回复必须包含：

- 文件结构树
- 核心交付物列表
- 覆盖率摘要
- 当前交付深度；分期交付时附深化任务清单摘要和尚未适用的红线
- 机械自检结果（错误清零；警告逐条说明处理方式）
- 状态看板路径与就绪统计（N/M 个模块可交开发、路径就绪情况）
- 缺陷处置清单（若本轮有协同评审或用户评审：修复/转排期/转开放问题三类对账，见「评审缺陷处置」）
- 高风险开放问题摘要
- 建议下一步评审顺序
