# ADR-0005：Logic Atlas 读模型

## 状态

已接受（设计方向，尚未实现）

## 日期

2026-07-20

## 摘要

PRD 工作树是写模型，Logic Atlas 是从权威主本确定性编译的只读投影，不建立第二套权威内容。正式逻辑关系只由确定性编译器 `build-logic-model.py` 写入，Agent 只产生候选和缺口；新鲜度由输入内容摘要机械判定并 fail closed；浏览器验证绑定渲染器版本并随插件 CI 执行，不绑定每次内容维护；说明性文案的自然化采用缓存与原文回退，不阻塞维护。本 ADR 由 ADR-0004 初版第 5–13 节拆分而来，并吸收 2026-07-20 多维度审查的定案。

## 背景

完整 PRD 同时承担需求审计、研发测试规格和人类理解产品逻辑三种职责。两级 `full-prd.md` 适合归档、全文检索和集中评审，但随着模块增加，正文拼接会自然变长。读者想理解页面关系、数据传递、模块依赖、状态变化和边界处理时，需要在模块 PRD、领域规格和全局文档之间来回跳转。继续压缩汇总 PRD 会损失可追溯性；再人工维护一份「精简 PRD」则会制造新的权威源。

审查确认了初版设计的两个问题，本版一并定案：

- 逻辑模型的提取者未定。如果由模型提取，下游的确定性校验只能挡住「编造的引用」，挡不住「漏提取的关系」，fail closed 的保证力名不符实。定案为确定性编译器，前置条件是模块模板补机器可解析契约。
- 收尾不变式把 Atlas 全量重建、模型调用（自然化）和浏览器验证绑成一个原子动作，日常维护成本过高，且让「浏览器是否安装」错误地决定 PRD 能否完成维护。定案为内容门与呈现门分层。

## 决策

### 1. PRD 工作树是写模型，Logic Atlas 是读模型

模块 `prd.md`、领域规格和全局主本继续保存完整事实、裁决和验收约束。Logic Atlas 把同一组主本投影成面向人阅读的逻辑视图，不建立第二套权威内容。

Logic Atlas 聚焦：

- 系统、模块和页面导航关系。
- 页面入口、角色、前置条件、用户动作和系统结果。
- 数据读取、写入、权威来源、同步方式和一致性边界。
- 跨模块、跨系统调用与失败传播。
- 端到端流程、业务对象状态变化和关键分支。
- 权限、租户、套餐和数据可见范围。
- 加载、空数据、弱网、超时、重试、重复提交、并发、对象不存在和状态已变化等边界。
- 补偿、幂等、日志与通知。
- 逻辑缺口、开放问题、需求生命周期（含废弃链路与迁移关系）和反向变更影响。

Atlas 不复述业务背景、原始需求全文和完整追溯表。读者需要审计时沿来源链接返回主本。

### 2. 一个逻辑模型，一份全局 Atlas，多种作用域

默认生成一个全局机器模型和一个全局 HTML，不与根 `full-prd.md`、系统 `*-full-prd.md` 一一维护独立 Atlas。

```text
prd/
├── logic-atlas.html
└── _generated/logic/
    ├── logic-model.json
    ├── manifest.json
    ├── logic-atlas.md
    └── validation-report.md
```

- `logic-model.json` 是一次构建内所有阅读视图的唯一机器模型，但仍是可删除、可重建的生成物。
- `logic-atlas.html` 是全局入口，通过系统、模块、页面和流程作用域过滤内容。
- `logic-atlas.md` 是同模型生成的文本降级版。
- 系统 README 可以链接到全局 Atlas 的对应作用域。
- 如果交付场景需要独立系统 HTML，只能在同一次构建中从同一个 `logic-model.json` 生成，不允许独立维护或单独补写内容。

两级 `full-prd.md` 是正文聚合视图，Logic Atlas 是关系投影视图。两者维度不同，不建立一一对应关系。

### 3. 确定性逻辑模型编译器

正式逻辑关系的唯一写入者是确定性编译器：

- 实现位置：`prd-from-requirements/scripts/build-logic-model.py`，调用方为 `prd-maintain` 和 `prd-sync`。
- 输入：模块 `prd.md`、领域规格、跨系统流程与依赖、权限与功能开通矩阵、追溯矩阵和开放问题清单；需求身份消费 `_generated/requirements-ledger.jsonl`（见 ADR-0004），保持「矩阵 → ledger → 编译器」单一解析路径，不对追溯矩阵做第二套解析。Ledger 是受验证的中间生成物：编译前必须从权威输入重建并验证其输入摘要（见第 6 节），它可以作为编译依赖，但不是权威事实来源。
- 输出：`_generated/logic/logic-model.json`。

职责分层：

1. 编译器机械提取结构化事实：需求、页面、流程、状态、字段、权限、模块关系、来源引用。
2. 无法从标准章节或表格确定的内容标记为「缺失/不可解析」，不猜测。
3. Agent 可以生成候选说明或指出缺口，但不能直接写入正式逻辑关系。
4. `text-naturalizer` 只处理模型中已验证的 narrative 文案，不参与提取和判断。

`logic-model.schema.json` 是公共契约，按 ADR-0002 的公共接口口径管理版本；schema 变更触发渲染器重新验证（见第 8 节）。

前置条件：现有模块 PRD 结构不完全统一（验收级符合完整模板，骨架级可能只有自由文本和需求区间），第一版编译器不能凭空准确提取所有页面与数据流。实施顺序固定为：先定义 schema，在模块模板中补机器可解析的页面契约、数据读写和模块交互表，对旧工作树执行一次结构迁移（无法确认的内容进缺口清单），编译器只消费明确结构，最后才生成 Atlas。

### 4. Atlas 只读取权威主本

逻辑模型直接读取：模块 `prd.md`、`00-global/domain-specs/*.md`、产品总览、跨系统流程和依赖、功能开通矩阵、开放问题、需求追溯矩阵（身份经由 ledger）。

逻辑模型不读取：根 `full-prd.md`、系统 `*-full-prd.md`、状态看板、上一次生成的 Logic Atlas 或逻辑模型、Excel 转换后的 Markdown。这样避免生成物依赖生成物、重复解析和内容放大。唯一例外是 `requirements-ledger.jsonl`：它是经新鲜度验证的中间生成物（第 6 节），允许作为编译依赖；其余生成物一律不读取。构建前先检查 ADR-0004 的读取栅栏：存在未完成发布的 operation 时，先完成幂等恢复或拒绝构建，不读取半发布状态的主本。

### 5. 每条 Atlas 内容必须携带来源

逻辑模型中的节点、关系和规则必须保存来源文件、章节或稳定锚点、需求编号以及可选的 OQ 编号。信息状态至少区分：原始需求、已裁决、推断、推荐默认和缺失。

没有来源的内容不得进入正式逻辑结论，只能作为逻辑缺口或待确认项展示。模块仍为骨架级，或主本没有定义页面、数据权威源、异常处理时，Atlas 显示「缺失」「待深化」或关联 backlog/OQ，不根据常见产品经验补齐。生成器以暴露缺口为目标，不以画出完整图为目标。

### 6. 用输入摘要保证新鲜度

`manifest.json` 保存生成器版本、`logic-model.schema.json` 版本、自然化规则版本、全部输入主本的路径和 SHA-256，以及三个摘要：

- `authoritativeSourceDigest`：排序后的「相对路径 + 文件哈希」计算出的权威主本摘要。
- `ledgerSourceDigest`：Ledger「当前有效输入集合」（ADR-0004 第 4 节：各源 `appliedRevision` 的 `normalized.jsonl`、已生效 change manifest、已越过提交点的 operation 的裁决 segment、`source-registry.json` 与 sync-state 的 applied 游标投影、追溯矩阵）的同法摘要。pending revision 与 pending change 不纳入——未确认内容不得提前进入 Atlas，观测到新版本也不得让 PRD 未变的 Atlas 无故过期；它们属于独立的 candidate overlay。
- `ledgerArtifactDigest`：编译实际消费的 ledger 文件哈希。

每次检查重新计算当前权威主本与 Ledger 输入的摘要：三者全部一致，Atlas 内容对应当前输入；任一不一致（包括旧 Ledger 配新主本），Atlas 标记过期并使机械检查失败；manifest 引用不存在的文件、漏掉应纳入的模块或包含 ledger 之外的生成物，检查失败。不依赖文件修改时间——移动、回滚和跨机器复制都以内容哈希判断。

### 7. 模型校验采用 fail closed 质量门

渲染前必须通过逻辑模型校验：

- 模型符合 `logic-model.schema.json`。
- 每个页面和动作归属已存在的模块。
- 每个需求编号存在于 ledger；生命周期为 `withdrawn`/`removed` 的需求不出现在当前视图的有效路径中（历史与迁移关系默认折叠、可切换查看，不隐藏）。
- 每条跨模块关系的两端都存在。
- 流程引用的状态存在于相应状态机；无法机械判断时产生人工复核项。
- 数据对象链接到领域规格或显式标记「权威源未确认」。
- 工作树模块、端到端流程与 Atlas 覆盖数量对账。
- 验收级模块的页面、流程、边界、权限和数据视图达到规定覆盖门槛。
- 来源文件和稳定锚点可解析；不存在重复 ID、孤立引用或循环空引用。

校验失败时不产出「最新可用」的成功状态，也不以旧版 Atlas 冒充当前结果。operation 校验失败时旧 Atlas 仍对应上一次有效视图，不算过期；HTML 永远自述生成快照摘要，不静态宣称「当前最新」。带外修改导致摘要过期时，只读检查器返回失败并报告 stale；替换入口的失败/过期页由显式写入 operation 产生，检查器不写任何文件。

机械覆盖与内容成熟度分开展示。存在一张表只说明结构可解析，不代表逻辑正确；Atlas 顶部同时展示覆盖数、缺失数、未解析引用、关联 OQ、当前深度和机械信号，避免可疑绿灯。fail closed 挡住的是编造的引用，挡不住主本本身的内容缺陷——后者依赖 PRD 侧的独立核验流程。

### 8. 渲染、静态检查与渲染器验证证明

HTML/Markdown 由已验证的渲染器确定性生成。每次构建后执行静态检查：数据完整、内容已转义、内部路由与来源链接有效、图形渲染失败时保留表格或列表降级内容。

浏览器级验证绑定「渲染器变化」，不绑定「PRD 内容变化」。需要浏览器验证的场景：

- HTML 模板、CSS 或 JavaScript 改变。
- `logic-model.schema.json` 改变。
- 图形布局算法或路由逻辑改变。
- Atlas 生成器版本升级。
- 用户显式要求执行可视化 QA。
- 插件发布前 CI。

浏览器验证内容：无脚本和控制台错误；来源链接、作用域路由、搜索和过滤可用；大型关系图支持聚焦、折叠和分层；常用桌面宽度下无不可控横向滚动和文字遮挡；键盘可导航，颜色不是唯一状态表达；固定样例执行截图或 DOM 快照回归。

验证通过后生成渲染器验证证明：

```json
{
  "rendererVersion": "1.0.0",
  "generatorVersion": "1.0.0",
  "schemaVersion": "1",
  "assetDigest": "sha256:...",
  "fixtureDigest": "sha256:...",
  "validationHarnessVersion": "1.0.0",
  "browserMatrixVersion": "2026-07",
  "browserValidated": true,
  "validatedAt": "2026-07-17T18:00:00+08:00"
}
```

验证证明随插件资产在 CI 中产生和发布，不随工作树维护。只要 `rendererVersion`、`generatorVersion`、`schemaVersion`、`assetDigest`、`fixtureDigest`、`validationHarnessVersion`、`browserMatrixVersion` 全部未变，内容更新继承该验证结果，不重新打开浏览器；fixture 或验证 harness 更新同样使旧证明失效。用户机器原则上永远不需要浏览器：内容维护正常完成，Atlas 展示「生成快照摘要及渲染器验证状态」；只有渲染器本身变化且没有对应验证证明时，才阻止插件发布或新渲染器投入使用。

### 9. 受限自然化：缓存与原文回退，不阻塞维护

Logic Atlas 面向人阅读，说明性正文按 `text-naturalizer` 的技术文档规则做受限自然化。自然化属于呈现质量，不属于逻辑正确性，不是维护的阻塞项：

- 已有缓存且来源片段未变化：直接复用。
- 新增或变化的 narrative：默认先使用未经自然化的原文——即从主本说明段落确定性摘录、带来源锚点的文本，保证 `copyPolished` 为否时内容仍全部可溯源。
- 模型可用时可在本轮增量自然化并更新缓存；不可用或校验失败时回退原文，只产生「文案未润色」提示。
- 显式执行 `prd-atlas-polish` 或发布前整理时批量补齐。

允许处理：页面、流程和模块的简短说明；数据流、依赖关系和边界场景的自然语言解释；缺口、风险和开放问题的背景说明；空状态、错误提示、筛选说明等固定界面文案（固定文案在渲染模板中一次写好并按规则评审，不逐次生成）。

禁止处理：需求编号、验收编号、OQ 编号和稳定节点 ID；状态名、角色名、字段名、页面名、模块名和领域术语；数字、金额、时间、枚举、条件表达式和布尔规则；权限、状态流转、异常分支、补偿、幂等和一致性约束；原始需求摘录、来源路径、章节锚点和哈希；表格、关系边、机器标签和其他结构化数据。

缓存键为「来源片段哈希 + 规则版本 + 模型标识」；`manifest.json` 记录自然化规则版本，规则升级可触发文案重建，但不能改变来源摘要和逻辑关系。

自然化后的复核分两层：

- 机械项（fail closed）：编号、术语、数值和来源引用集合与处理前逐项对账一致。
- 语义项（抽查复核，不宣称机械保证）：没有新增事实、默认值、因果关系或范围判断；没有把「缺失」「待确认」「推荐默认」写成已确定结论；没有弱化「必须」「禁止」「仅当」等约束强度。

任一检查失败，丢弃自然化结果并回退原文。`text-naturalizer` 只能改善表达，不能成为产品判断或逻辑补全工具。

### 10. 内容门与呈现门

`prd-maintain` 与 `prd-sync` 的收尾分两层。

**内容门（每次维护阻塞）**：

1. 把模块与全局主本修改写入 operation staging（不改原路径）。
2. 重生成两级汇总 PRD。
3. 运行 PRD 工作树机械自检。
4. 按 `sync-state.json` 的 applied 游标选取当前有效输入集合，重建 `requirements-ledger.jsonl` 并验证 `ledgerSourceDigest`（不扫描整个 `revisions/` 目录，pending 不参与）。
5. 确定性重建 `logic-model.json`（只消费已验证的 Ledger）。
6. 检查 schema、来源、引用、覆盖与三个摘要（第 6 节）。
7. 用已验证的渲染器确定性生成 HTML 与 Markdown 视图（narrative 按第 9 节缓存或原文回退）。
8. 执行静态 HTML 检查。
9. 重生成状态看板并复核摘要。

以下失败阻塞维护完成：三个摘要任一落后（含旧 Ledger 配新主本）；schema 或引用检查失败；正式逻辑没有来源；HTML 生成失败；静态检查发现缺少数据、未转义内容或内部路由无效。

内容门按已交付阶段裁剪：第一阶段为 Markdown-only 内容门，交付物是 `logic-atlas.md`，第 7–8 步的 HTML 部分不适用；HTML 生成与静态检查随第二阶段渲染器发布启用，启用后成为无条件阻塞项。不允许出现「阶段一工作树因缺少 HTML 能力而无法完成维护」的状态。

维护会话中，第 2–9 步全部针对 ADR-0004 定义的「预提交验证视图」（operation overlay）执行：机械自检以 `--operation-id` 检查暂存内容；两级汇总、Ledger、逻辑模型、Atlas、manifest、validation report 与状态看板全部生成到 staging，并与主本修改一起列入发布清单。全部检查通过后逐文件原子发布，全部文件发布完成后才推进唯一提交点、再补写最终状态；检查失败或发布未完成时提交点不推进，当前有效视图与旧 Atlas 保持上一次成功提交的状态。只让部分步骤走验证视图会出现「新主本通过旧树检查」或「主本已提交、Atlas 仍是旧版本」。

**呈现门（不阻塞日常维护）**：自然化补齐绑定 `prd-atlas-polish` 与发布整理；浏览器验证绑定渲染器版本与插件 CI（第 8 节）。

Atlas 状态不压成一个布尔值，分维度报告：

| 状态 | 含义 | 是否阻塞日常维护 |
|---|---|---|
| `contentFresh` | 对应当前 PRD 输入摘要 | 是 |
| `structurallyValid` | schema、来源、引用检查通过 | 是 |
| `renderSucceeded` | HTML/Markdown 成功生成 | 是 |
| `copyPolished` | 说明文案已自然化 | 否 |
| `presentationVerified` | 当前渲染器通过浏览器 QA | 日常维护否，渲染器发布是 |
| `presentationRisk` | 超长文本、超大图等呈现风险 | 警告，按需验证 |

成本基准：只定案一条 OQ 时，收尾为回扫引用 → 更新主本 → 重生成汇总 → 确定性重建模型和当前阶段已启用的视图 → 静态检查 → 更新看板，不调用 `text-naturalizer`、不启动浏览器，分钟内完成。Atlas 编译与渲染控制在秒级；超过性能预算（如 5 秒）才做模块级增量编译，不预先引入增量缓存。

## 备选方案

### 人工维护一份精简 PRD

短期阅读方便，但会形成与模块主本竞争的第二权威源，且没有稳定的新鲜度判断。拒绝；阅读视图必须可删除、可重建。

### 为每个 `full-prd.md` 维护独立 Atlas

文件对应关系直观，但跨系统流程会重复，系统边界移动时容易产生不同版本。拒绝独立维护；如需独立文件，只允许同源、同批次物化。

### 直接让模型从 PRD 自由总结 HTML

开发成本低，但内容不可重复、难以校验来源，也可能为了完整性补写不存在的逻辑。拒绝；模型可协助解释或产出候选，正式结果必须经确定性编译与校验。

### 模型辅助提取正式逻辑关系

省掉模板结构改造，但下游校验只能验证引用完整性，无法验证提取完整性——漏提取的关系能通过全部质量门，fail closed 的承诺落空。拒绝；正式关系只由确定性编译器写入，代价是模板机器可解析化与存量迁移前置。

### 收尾强绑定自然化与浏览器验证（ADR-0004 初版第 13 节）

保证「Atlas 永远新鲜且已验证」，但把最重、环境依赖最强的环节放进最高频路径：一条 OQ 定案要付出模型调用与浏览器启动的成本，无浏览器的机器上维护永远无法宣告完成，实际效果是诱导绕流程。拒绝；确定性部分（编译、渲染、静态检查）留在阻塞路径，非确定性部分（自然化、浏览器）绑定各自真正的变化源。

## 影响

- 模块模板（`module-prd.md`）新增机器可解析的页面契约、数据读写和模块交互表；`check-prd-tree.py` 同步检查新结构章节，避免「模板要求了但没人验」。
- 旧工作树需要一次结构迁移；无法确认的内容进缺口清单，不伪造结构。Logic Atlas 不能提升源 PRD 的内容深度，初次生成可能暴露大量 backlog，这是预期结果。
- `logic-model.schema.json` 成为公共契约，变更需要版本管理并触发渲染器重新验证。
- 编译器与渲染器必须满足插件自包含与可移植性约束（Python 标准库、自包含前端资产）；浏览器验证成本移至插件 CI，用户机器无浏览器依赖。
- 自然化结果缓存避免无事实变化的整页措辞漂移；非确定性文案不污染结构化逻辑、不制造无意义 diff。
- `prd-maintain` 与 `prd-sync` 的收尾不变式按第 10 节内容门重写；新增 `prd-atlas-polish` 显式入口（命名按 ADR-0002 登记）。

## 分阶段落地

### 前置阶段：结构契约与存量迁移

- 定义 `logic-model.schema.json`。
- 模块模板补机器可解析的页面契约、数据读写和模块交互表；`check-prd-tree.py` 同步扩展。
- 对旧工作树执行结构迁移，无法确认的内容进缺口清单。

### 第一阶段：逻辑模型与 Markdown 视图

- 实现 `build-logic-model.py`，只消费明确结构，消费 ledger 获取需求身份。
- 实现 schema、来源、引用、覆盖、三摘要新鲜度与 fail closed 检查。
- 生成 `logic-atlas.md` 文本视图；以 Markdown-only 内容门接入收尾（HTML 步骤随第二阶段启用）。

### 第二阶段：HTML 与渲染器验证

- 实现自包含 HTML 渲染器与作用域入口、静态检查。
- 建立 fixture 回归与插件 CI 浏览器验证，产出渲染器验证证明。
- Atlas 六维状态展示与看板对接。

### 第三阶段：受限自然化

- narrative 字段接入受限 `text-naturalizer`，实现缓存、机械对账与原文回退。
- 新增 `prd-atlas-polish` 批量补齐入口。

### 第四阶段：按主题解释

- 增加面向页面、流程或模块的按需解释入口。
- 默认只在会话中返回，不落盘制造长期副本。
- 用户要求保存时，产物绑定 `authoritativeSourceDigest`；源摘要变化后明确标记过期。

## 验收原则

- 任一权威主本变化后，旧 Atlas 必须被机械识别为过期；manifest 引用缺失文件、漏纳模块或包含 ledger 之外的生成物时检查失败。
- 旧 Ledger 与新主本的组合不得产出 `contentFresh` 的 Atlas：`ledgerSourceDigest` 或 `ledgerArtifactDigest` 不一致时内容门失败。
- pending revision 或 pending change 不改变 Atlas 新鲜度判定，未确认内容不出现在 Atlas 中。
- 第一阶段工作树以 Markdown-only 内容门完成维护，不因缺少 HTML 能力而失败；第二阶段启用后 HTML 步骤成为无条件阻塞项。
- Atlas 中每条正式逻辑都能回到存在的来源文件和需求编号或裁决；Agent 产出的候选内容不出现在正式逻辑关系中。
- 骨架级模块缺失的逻辑显示为缺口，不被自动补齐。
- 全局与系统作用域来自同一个逻辑模型，不出现独立版本。
- schema、来源、引用或静态 HTML 检查失败时，维护流程不能宣告内容门通过。
- 单条 OQ 定案的收尾在分钟内完成；Atlas 编译与渲染秒级；超预算才引入模块级增量编译。
- 自然化后编号、术语、数值、来源集合与处理前一致，不满足时自动回退原文；自然化不可用时维护正常完成，仅提示文案未润色。
- PRD 内容变化不触发浏览器验证；渲染器、schema 或前端资产变化且无对应验证证明时，插件不能发布、新渲染器不能投入使用。
- Atlas 状态按六个维度分别展示，不以单一绿灯呈现。

## 明确不做

- 不允许人工编辑 Logic Atlas 作为修复手段。
- 不让 Logic Atlas、状态看板或汇总 PRD 成为其他生成物的权威输入。
- 不用模型推断补齐源 PRD 未定义的页面、数据和异常规则；模型不直接写入正式逻辑关系。
- 不把整份 HTML 交给模型自由重写，也不在每次构建时重新生成固定界面文案。
- 不在日常内容维护路径中启动浏览器或强制模型调用。
- 不预先引入增量编译缓存；不为每个汇总 PRD 维护独立 Atlas。
